 反馈校验规则

 反馈帧使用 1 字节累加和校验：

 CHECKSUM = (A1 + A2 + XL + XH + YL + YH + ZL + ZH) & 0xFF

 注意校验不包含：

 AA BB

 Byte10

 CHECKSUM 自身

 代码位置：src/car_bringup/src/chassis_feedback_node.cpp:81

 反馈速度解码

 反馈的速度字段同样按低字节在前、高字节在后读取：

 x_raw = XH << 8 | XL

 y_raw = YH << 8 | YL

 z_raw = ZH << 8 | ZL

 然后发布到 /vel_raw：

 linear.x  = x_raw / 1000.0

 linear.y  = y_raw / 1000.0

 angular.z = z_raw / 1000.0

 代码位置：src/car_bringup/src/chassis_feedback_node.cpp:86

 例如底盘反馈：

 AA BB 0A 12 F4 01 00 00 00 00 00 21

 其中：

 A1+A2+XL+XH+YL+YH+ZL+ZH

 = 0x0A + 0x12 + 0xF4 + 0x01 + 0 + 0 + 0 + 0

 = 0x107

 & 0xFF = 0x07

 这个例子的 checksum 应该是 07，不是 21。正确帧应为：

 AA BB 0A 12 F4 01 00 00 00 00 00 07

 它会被解析为：

 linear.x = 0.5 m/s

 linear.y = 0.0 m/s

 angular.z = 0.0 rad/s

 重要实现细节：负数反馈解析

 当前反馈解码函数在 src/car_bringup/src/chassis_feedback_node.cpp:17：

 if (high_byte & 0x80) {

   return 0 - (65535 - value);

 }

 这不是标准二进制补码解码。标准写法通常应是：

 value - 65536

 当前写法对负数会有 1 个计数的偏差。例如：

 0xFFFF 标准应为 -1

 当前代码解为 0

 0xFE0C 标准应为 -500

 当前代码解为 -499

 因为最后还会除以 1000.0，所以偏差是 0.001 单位量级。这个行为在旧 Python 版 src/car_bringup/scripts/pubv.py:11 里也一样，说明这是从旧协议实现继承来的行为。

 如果底盘反馈严格按标准 int16 补码发送，当前工程的负速度反馈会有 0.001 的偏差。

 协议摘要

 当前工程与底盘的协议可以总结为：

 物理层：

  UART 串口，默认 /dev/carserial，115200 8N1，raw，无流控

 控制输入：

  ROS /cmd_vel，geometry_msgs/Twist

 发送帧：

  12字节二进制

  AA BB 0A 12 02 XL XH YL YH ZL ZH 00

 发送速度编码：

  int16(value * 1000)

  小端序，低字节在前

  linear.x -> X

  linear.y -> Y

  angular.z -> Z

 初始化帧：

  11 00 00 00 00 00 00 00 00

 停止帧：

  AA BB 0A 12 02 00 00 00 00 00 00 00

 反馈帧：

  12字节二进制

  AA BB A1 A2 XL XH YL YH ZL ZH ? CHECKSUM

 反馈校验：

  CHECKSUM = (A1 + A2 + XL + XH + YL + YH + ZL + ZH) & 0xFF

 反馈校验规则

 反馈帧使用 1 字节累加和校验：

 CHECKSUM = (A1 + A2 + XL + XH + YL + YH + ZL + ZH) & 0xFF

 注意校验不包含：

 AA BB

 Byte10

 CHECKSUM 自身

 代码位置：src/car_bringup/src/chassis_feedback_node.cpp:81

 反馈速度解码

 反馈的速度字段同样按低字节在前、高字节在后读取：

 x_raw = XH << 8 | XL

 y_raw = YH << 8 | YL

 z_raw = ZH << 8 | ZL

 然后发布到 /vel_raw：

 linear.x  = x_raw / 1000.0

 linear.y  = y_raw / 1000.0

 angular.z = z_raw / 1000.0

 代码位置：src/car_bringup/src/chassis_feedback_node.cpp:86

 例如底盘反馈：

 AA BB 0A 12 F4 01 00 00 00 00 00 21

 其中：

 A1+A2+XL+XH+YL+YH+ZL+ZH

 = 0x0A + 0x12 + 0xF4 + 0x01 + 0 + 0 + 0 + 0

 = 0x107

 & 0xFF = 0x07

 这个例子的 checksum 应该是 07，不是 21。正确帧应为：

 AA BB 0A 12 F4 01 00 00 00 00 00 07

 它会被解析为：

 linear.x = 0.5 m/s

 linear.y = 0.0 m/s

 angular.z = 0.0 rad/s

 重要实现细节：负数反馈解析

 当前反馈解码函数在 src/car_bringup/src/chassis_feedback_node.cpp:17：

 if (high_byte & 0x80) {

   return 0 - (65535 - value);

 }

 这不是标准二进制补码解码。标准写法通常应是：

 value - 65536

 当前写法对负数会有 1 个计数的偏差。例如：

 0xFFFF 标准应为 -1

 当前代码解为 0

 0xFE0C 标准应为 -500

 当前代码解为 -499

 因为最后还会除以 1000.0，所以偏差是 0.001 单位量级。这个行为在旧 Python 版 src/car_bringup/scripts/pubv.py:11 里也一样，说明这是从旧协议实现继承来的行为。

 如果底盘反馈严格按标准 int16 补码发送，当前工程的负速度反馈会有 0.001 的偏差。
